Entities

  • All entities should be geocoded and available for mapping. In the event that an address is a Post Office Box or cannot be displayed for other reasons, the geocoding should be set to City, State (e.g. Detroit, MI).
  • All entities should have:
    • Visibility: Live (available to the front end) and Pending (not available to the front end)
    • Administrator Flag: True or False; marks an entity as requiring admin attention
  • All entities should be marked with one or more of the following disciplines:
    • Music
    • Dance
    • Film & Electronic Media
    • Visual Arts
    • Theatre
    • Literary Arts
    • Humanities
  • All entities might be marked with one or more of the following primary disciplines:
    • Music
    • Dance
    • Film & Electronic Media
    • Visual Arts
    • Theatre
    • Literary Arts
    • Humanities (Humanities is a special case - if an entity is categorized as "Humanities," it should automatically be categorized as "Primarily Humanities.")
  • All entities must be one of the following types:
    • Non-profit
    • For-profit
    • Unincorporated
    • Individual Artist
    • Government
Guidestar API (Non-Profit Basic Info)
Gemini SBS authored a Ruby gem to interact with the Guidestar API. Github link
Guidestar API Credentials
  • nationwide data source
  • should begin building historical information from this source beginning in 2012
  • Imported data is first passed through a script that makes an initial attempt to classify organizations, then each organization in Match Scores 3-5 is goes through Amazon Mechanical Turk processing to classify disciplines and activity. Current non-profit data in both Detroit and San Francisco has undergone both processes.
  • The application should pull from the API once per year. All new organizations should be exported to be put through the Mechanical Turk process.

Detroit Organizations - data set not yet acquired (spreadsheet)
San Francisco Organizations - data set not yet acquired (spreadsheet)

Cultural Data Project (CDP - Extended Non-Profit Info)
Link to CSV - Detroit Organizations
San Francisco Organizations - data set not yet acquired (spreadsheet)
  • All data from linked CSV should be stored.
  • There is currently a CDP table in BACAM that may be helpful; however, the fields have changed significantly, and all data (both for Detroit and San Francisco) will need to be re-imported.
  • The CDP extended info will be connected with an organization's main profile via its EIN (Employer Identification Number). In the CDP data set, this field is called "federal_id".
  • In the database, the CDP information will remain separate from the Guidestar/NCCS non-profit information.
  • In the interface, the CDP information should replace certain fields when available, as the information is more accurate than the information from Guidestar. (Please see this spreadsheet for all fields that should be replaced in the interface, as well as the equivalent field names and notes.)
  • If an organization is in this data set and not in our database, it should be imported from Guidestar.
InfoUSA (For-profit Info)
Link to CSV - Detroit businesses
San Francisco businesses - data set not yet acquired (spreadsheet)
  • The API should deliver these to the front end of the application separately from the non-profit businesses. A user might want to see *all* organizations (both for- and non-profit), or they might want to see only for-profit businesses and/or non-profit businesses.
  • Import Logic
    • Any time one of the seven FA disciplines is listed in either column, we categorize the business as that discipline. (This will limit the highest number of possible disciplines for each business to two.)
    • All other words that appear except "other" should be added to the business as a tag (e.g. "library, religious, arts education, multidisciplinary, etc.").
    • If the business does not have any Fractured Atlas disciplines, set the entity to visibility: pending and flag for an administrator.
Fractured Atlas API (Fiscally Sponsored Projects)

Gemini SBS built a rake task (called fiscal_sponsors.rake, in the pa_data repo) that downloads the data from the FA database to the Archie database.

  • All fiscally-sponsored projects should be marked as "unincorporated."
  • The Archipelago software should be able to access the Fractured Atlas fiscal sponsorship database and sync project records on a regular basis.
  • When the Archipelago software creates a record for a Fractured Atlas fiscally sponsored project, certain organization attributes are set automatically
    • All FA sponsored projects are primarily arts.
    • All FA sponsored projects are "unincorporated" entities.
    • All sponsored projects have 1 to n fiscal sponsors. All FA sponsored projects receive Fractured Atlas's organization ID as a fiscal sponsor.
    • Active/Inactive FA sponsored projects should set the respective inactive flag in Archie
  • A fiscally sponsored project should have their geocode defaulted to the city and state if they have not opted in to share their "full profile"
    • The projects will be geocoded according to their actual address in the Fractured Atlas database. However, their exact adddress and location will not be revealed to the user. Instead, they will be assigned to the city in which they are located, much the same way that P.O. Boxes are handled.
