Entities

  • All entities should be geocoded and available for mapping. In the event that an address is a Post Office Box or cannot be displayed for other reasons, the geocoding should be set to City, State (e.g. Detroit, MI).
  • All entities should have:
    • Visibility: Live (available to the front end), Pending (not available to the front end), and Inactive (not available to the front end)
    • Administrator Flag: True or False; marks an entity as requiring admin attention
    • Administrator Flag Comment: should provide a reason for the admin flag
  • All entities should be marked with one or more of the following disciplines:
    • Music
    • Dance
    • Film & Electronic Media
    • Visual Arts
    • Theatre
    • Literary Arts
    • Humanities
  • All entities might be marked with one or more of the following primary disciplines:
    • Music
    • Dance
    • Film & Electronic Media
    • Visual Arts
    • Theatre
    • Literary Arts
    • Humanities (Humanities is a special case - if an entity is categorized as "Humanities," it should automatically be categorized as "Primarily Humanities in the front end.")
  • All entities must be one of the following types:
    • Non-profit
    • For-profit
    • Projects
    • Individual Artist
    • Government
  • All individual entities (of all types) should have a list of tags. (Associated user stories in Trello: a user should be able to tag an entity; tags should be searchable.)
  • All entity types should be able to be delivered to the front end of the application separately from one another. For instance, the front end should be able to get a list of *only* non-profit entities, as well as be able to call a list that includes *both* for-profit and project entities.
  • Users may add entities to the system and edit entities they have claimed. (See section "User Types" for more details)
    • Entities of the following types may be added or claimed: nonprofit, for-profit, government, or individual artist.
    • Project entities will never be edited or claimed by users.
    • See Trello story for details (including attached wireframes).
    • Once a user is associated with any entity, that user may edit some data for the entity. For a complete list of user-editable fields, see column D in the entities schema.
Importing Entities
  • Funding data sets, Spaces, and the Fractured Atlas fiscal sponsorship API are secondary data sources for entities. (Meaning: any entities that we find in those data sources should be added to the database.)
  • When we come across an entity in one of these data sources that already exists in the Archipelago database:
    • Do not override any fields with data in them.
    • If the data source provides data for fields that are NULL, store the new data.
  • When we come across an entity in one of these data sources that does NOT exist in the Archipelago database:
    • Add an entity record with the information we acquire from the source (e.g. organization name).
    • Search Guidestar for the entity. If a match is found, store the data and classify the entity as a nonprofit.
    • Mark the entity "pending."
    • Flag the entity for admin review. (Add comment to flag: "Imported from $datasource")
    • All new non-profit entities will need to be exported and run in batches through Mechanical Turk processing.
    • All new entities added through this process will be looked at by a SA admin, coded properly, saved, and marked "active."
  • Some fields should never be overwritten when importing new data because they were created manually. These are in the column "Manually Written?" in the Entities Data Schema.

  • Non-profit - historical data for non-profits comes from two places:
    • NCCS provides the main historical record (see info below)
    • CDP also provides historical information on many non-profits (see info below)
  • For-profit - start building a historical record per business from current year (no back-data)
  • Government - start building a historical record per business from current year (no back-data)
  • Project - no historical record
  • Individual Artist - no historical record
     
Guidestar API (Non-Profit Basic Info)
Gemini SBS (the consultants who built BACAM) authored a Ruby gem to interact with the Guidestar API. Github link
Guidestar API Credentials
  • nationwide data source for non-profit entities
  • should begin building historical information from this source beginning in 2012
  • We pull organizations from Guidestar based on
    1. the instance region (see counties for Detroit and Bay Area instances here)
    2. a script that pulls and classifies organizations according to some criteria. (Script documentation in progress; it's built in to BACAM already.) On import, data is first passed through a script that makes an initial attempt to classify organizations, then each organization in Match Scores 3-5 is goes through Amazon Mechanical Turk processing to classify disciplines and activity. Current non-profit data in both Detroit and San Francisco has undergone both processes, so this data needs to be migrated from the old databases.
  • The application should pull from the API once per year. All new organizations should be exported and sent to Alison Conard to be put through the Mechanical Turk process.
  • All newly-imported organizations should be set to visibility: pending and marked for inclusion in the next Mechanical Turk batch.
  • Any time a non-profit organization exists in a data source other than Guidestar, we need to pull that organization from Guidestar, store it, and it needs to be run through the Mechanical Turk process.
  • Examples of Mechanical Turk files here; includes input files (input into M Turk), results (export from M Turk), and Archie output (after M Turk results were imported into database)

Fractured Atlas API (Fiscally Sponsored Projects)

Gemini SBS built a rake task (called fiscal_sponsors.rake, in the pa_data repo) that downloads the data from the FA database to the Archie database. This should contain the necessary credentials to access the fiscal sponsorship database.

The API will be undergoing changes soon and will provide financial data (a "full profile"), in addition to the data it currently provides.

This data should be pulled once/week.

  • All fiscally-sponsored projects should be classified as "project."
  • The Archipelago software should be able to access the Fractured Atlas fiscal sponsorship database and sync project records on a regular basis.
  • When the Archipelago software creates a record for a Fractured Atlas fiscally sponsored project, certain organization attributes are set automatically
    • All FA sponsored projects are primarily arts.
    • All FA sponsored projects are "unincorporated" entities.
    • All sponsored projects have 1 to n fiscal sponsors. All FA sponsored projects receive Fractured Atlas's organization ID as a fiscal sponsor.
    • Active/Inactive FA sponsored projects should set the respective inactive flag in Archie
  • A fiscally sponsored project should have their geocode defaulted to the city and state if they have not opted in to share their "full profile"
    • The projects will be geocoded according to their actual address in the Fractured Atlas database. However, their exact adddress and location will not be revealed to the user. Instead, they will be assigned to the city in which they are located, much the same way that P.O. Boxes are handled.
  • For all Fractured Atlas fiscally sponsored projects, set Fractured Atlas Productions, Inc. as fiscal_sponsor_id. Fractured Atlas must be pulled from Guidestar so that it exists in the db as an entity. (This should happen w/ every new instance.) I believe you can pull w/ the EIN: 11-3451703. Spaces

Fractured Atlas Spaces API (places where art happens)
Spaces is another Fractured Atlas platform that lists places where art happens. See an example of a Spaces platform here.
  • API Sandbox Info
  • Bay Area Spaces data is integrated via this API into BACAM.
  • The API Sandbox Info document also contains the fields to store.
  • The Archie spaces table reflects the current contents of the Spaces database, so we overwrite all spaces on each import.
  • Each space in the spaces database is connected with an organization. We should match these organizations via name lookup with the organizations in our database so that our spaces table contains a foreign key to an entity. This means that we'll need a field that contains a foreign key link to the entities table. (I'll call it archie_entity_id for now.) The logic will be similar to finding organizations without EINs that we acquire in our grants data sources:
  1. For each organization in the Spaces API, look at the organization_name.
  2. Try to match the name to an organization already in our database. If there's a match, set "archie_entity_id" to the correct organization.
  3. For organizations that don't match, try to find a name match via the Guidestar API.
  4. If no match can be found, we'll need to figure out a way for an intern to match the grantee organizations w/ an organization EIN. Maybe a spreadsheet export of all grants that have not been matched to organizations that the intern can fill in?
 
 

User Types

Super Admin

  • Super admin users should be able to edit all data system wide. 
  • Super admins will be Fractured Atlas employees and affiliated developers (e.g. Xentio).
  • Super admins can create new admin users.

Instance Admin

  • Instance admins should have the same permissions as super admins, but only for a single instance (e.g. Detroit or Bay Area).
  • Instance admins will be partners in the particular implementation (e.g. Sustain Arts staff members).
  • Instance admins can be created only by super admins.

Users

  • User profiles will be created through a user sign-up process.
  • Users may choose to give us only their basic information (Email, first name, last name, zip code, password).
  • If a user completes the advanced user profile, we will also know what type of entity they work for and/or their role in the arts community.
  • User activation will happen automatically -- we will verify their email and their account will be active.
  • See schema for users here.
  • For a basic list of features surrounding user profiles and their permissions, see the SA Features (Dev) Trello board (User Profiles/Permissions list).
  • For a complete mockup of user profile (basic & advanced) creation, see: Wireframes for user profiles

Users' Relationships to Entities

  • Users may be associated with entities in the following ways:
  • For entity types nonprofit, for-profit, government:
    • A user can claim only one entity.
    • Multiple users can claim the same entity.
  • For entity type individual artist:
    • A user can claim one and only one individual artist profile that exists in our database.
    • Multiple users can *not* claim the same individual artist profile. If a user tries to claim an individual artist profile that has already been claimed by another user, the front end should offer the user a way to contact support and dispute the claim.
    • A user can claim *both* an individual artist profile and a business entity (nonprofit, for-profit, government, and project). (This is because users may be both an artist and work for an arts and culture entity.)
    • Users can also add entities to the database.
      • Like claiming entities, they may add a single business entity (nonprofit, for-profit, government, or project) and/or an individual artist profile.
      • Once the user adds an entity, the rules for claiming that entity are the same as above.
    • Users can never be associated with entities of type: project. (They cannot add or claim these entities.)
  • Once a user is associated with any entity, that user may edit some data for the entity. For a complete list of user-editable fields, see column D in the entities schema.

User Permissions

  • Only a logged in user can tag an entity.

User Types & Presets

  • A user has the option to create an advanced user profile.
  • If a user has an advanced profile, they will tell us the following:
  • Who they work for in the $entityregion arts community (they can choose one)
    • nonprofit organization
    • arts-related for-profit business
    • foundation
    • government
  • What is their role in the $entityregion arts community (they can choose multiple)
    • artist
    • arts educator
    • researcher/consultant/policy-maker
    • arts patron
    • individual donor
    • journalist/blogger
  • If the user chooses one of the bold entities, that will dictate which preset questions we display.
  • If the user does not choose a bold entity they work for, but DOES choose "artist," or "researcher/consultant/policy-maker," we will use the information to dictate which preset questions we display.
  • If the user chooses a role or entity type that is not bold, they see the default preset questions.
