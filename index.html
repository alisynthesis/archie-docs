Entities

  • All entities should be geocoded and available for mapping. In the event that an address is a Post Office Box or cannot be displayed for other reasons, the geocoding should be set to City, State (e.g. Detroit, MI).
  • All entities should have:
    • Visibility: Live (available to the front end) and Pending (not available to the front end)
    • Administrator Flag: True or False; marks an entity as requiring admin attention
  • All entities should be marked with one or more of the following disciplines:
    • Music
    • Dance
    • Film & Electronic Media
    • Visual Arts
    • Theatre
    • Literary Arts
    • Humanities
  • All entities might be marked with one or more of the following primary disciplines:
    • Music
    • Dance
    • Film & Electronic Media
    • Visual Arts
    • Theatre
    • Literary Arts
    • Humanities (Humanities is a special case - if an entity is categorized as "Humanities," it should automatically be categorized as "Primarily Humanities.")
  • All entities must be one of the following types:
    • Non-profit
    • For-profit
    • Unincorporated
    • Individual Artist
    • Government
  • All individual entities (of all types) should have a list of tags associated with it. (Associated user stories: a user should be able to tag an entity; a user should be able to call a list of entities w/ a particular tag.)
  • All entities should be able to be delivered to the front end of the application separately from one another. For instance, the front end should be able to get a list of *only* non-profit entities, as well as be able to call a list that includes *both* for-profit and unincorporated entities.
Guidestar API (Non-Profit Basic Info)
Gemini SBS (the consultants who built BACAM) authored a Ruby gem to interact with the Guidestar API. Github link
Guidestar API Credentials
  • nationwide data source for non-profit entities
  • should begin building historical information from this source beginning in 2012
  • We pull organizations from Guidestar based on
    1. the instance region (see counties for Detroit and Bay Area instances here)
    2. a script that pulls and classifies organizations according to some criteria. (Script documentation in progress; it's built in to BACAM already.) On import, data is first passed through a script that makes an initial attempt to classify organizations, then each organization in Match Scores 3-5 is goes through Amazon Mechanical Turk processing to classify disciplines and activity. Current non-profit data in both Detroit and San Francisco has undergone both processes, so this data needs to be migrated from the old databases.
  • The application should pull from the API once per year. All new organizations should be exported and sent to Alison Conard to be put through the Mechanical Turk process.
  • Any time a non-profit organization exists in a data source other than Guidestar, we need to pull that organization from Guidestar, store it, and it needs to be run through the Mechanical Turk process.

Cultural Data Project (CDP - Extended Non-Profit Info)
Link to CSV - Detroit Organizations
San Francisco Organizations - data set not yet acquired (spreadsheet)
  • This is a detailed data source that covers a subset of the total entities in the database (mostly non-profit organizations). All data are provided by the organizations themselves, so it is very accurate.
  • All data from linked CSV should be stored. (This data set is also a source for audience/participation data.)
  • There is currently a CDP table in BACAM that may be helpful; however, the fields have changed significantly, and all data (both for Detroit and San Francisco) will need to be re-imported.
  • The CDP extended info will be connected with an organization's main profile via its EIN (Employer Identification Number). In the CDP data set, this field is called "federal_id".
  • In the database, the CDP information should remain separate from the Guidestar/NCCS non-profit information.
  • In the interface, the CDP information should replace certain fields when available, as the information is more accurate than the information from Guidestar. (Please see this spreadsheet for all fields that should be replaced in the interface, as well as the equivalent field names and notes.) Note: this functionality will have to wait due to our license agreement w/ the CDP: no information about individual organizations may be shown in the interface.
  • If an organization is in this data set and not in our database, it should be imported from Guidestar.
  • The CDP has historical data (multiple years for the same entity) and should form part of the overall historical profile of the organizations included.
  • The CDP profiles also contain audience information, which is covered in more detail in the Audience section below.
InfoUSA (For-profit Entities Info)
Link to CSV - Detroit businesses
San Francisco businesses - data set not yet acquired (spreadsheet)
  • All for-profit entities should be classified as "for-profit."
  • All data from linked CSV should be stored.
  • We don't have an EIN for these businesses in the original data source. (Only an InfoUSA ID, which is not a federal ID.) This is okay, because we won't be connecting the for-profit businesses w/ any other data; we'll simply be displaying data about these businesses.
  • The API should be able to deliver these to the front end of the application separately from the non-profit businesses. A user might want to see *all* organizations (both for- and non-profit), or they might want to see only for-profit businesses and/or non-profit businesses.
  • Import Logic
    • Any time one of the seven FA disciplines is listed in either column, we categorize the business as that discipline. (This will limit the highest number of possible disciplines for each business to two.)
    • All other words that appear except "other" should be added to the business as a tag (e.g. "library, religious, arts education, multidisciplinary, etc.").
    • If the business does not have any Fractured Atlas disciplines, set the entity to visibility: pending and flag for an administrator.
Fractured Atlas API (Fiscally Sponsored Projects)

Gemini SBS built a rake task (called fiscal_sponsors.rake, in the pa_data repo) that downloads the data from the FA database to the Archie database. This should contain the necessary credentials to access the fiscal sponsorship database.

The API will be undergoing changes soon and will provide financial data (a "full profile"), in addition to the data it currently provides.

This data should be pulled once/week.

  • All fiscally-sponsored projects should be classified as "unincorporated."
  • The Archipelago software should be able to access the Fractured Atlas fiscal sponsorship database and sync project records on a regular basis.
  • When the Archipelago software creates a record for a Fractured Atlas fiscally sponsored project, certain organization attributes are set automatically
    • All FA sponsored projects are primarily arts.
    • All FA sponsored projects are "unincorporated" entities.
    • All sponsored projects have 1 to n fiscal sponsors. All FA sponsored projects receive Fractured Atlas's organization ID as a fiscal sponsor.
    • Active/Inactive FA sponsored projects should set the respective inactive flag in Archie
  • A fiscally sponsored project should have their geocode defaulted to the city and state if they have not opted in to share their "full profile"
    • The projects will be geocoded according to their actual address in the Fractured Atlas database. However, their exact adddress and location will not be revealed to the user. Instead, they will be assigned to the city in which they are located, much the same way that P.O. Boxes are handled.
  • For all Fractured Atlas fiscally sponsored projects, set Fractured Atlas Productions, Inc. as fiscal_sponsor_id. Fractured Atlas must be pulled from Guidestar so that it exists in the db as an entity. (This should happen w/ every new instance.) I believe you can pull w/ the EIN: 11-3451703. Link to Fractured Atlas on Guidestar website
Kickstarter (Kickstarter Projects)
Detroit Projects - acquired; needs cleaning
Detroit Yearly Metrics - acquired; needs cleaning
San Francisco Projects - acquired; needs cleaning
San Francisco Yearly Metrics - acquired; needs cleaning
  • All Kickstarter projects should be classified as "unincorporated" entities
  • Notes from KS about their data:
    • The total_pledges column in the yearly metrics CSVs represent the total number of pledges not unique backers, since many users back multiple projects. It is worth emphasizing this to Fractured Atlas so that they don't accidentally assume there are more backers to these projects than there are. 
    • I included live project money in the yearly metrics to be completist 
    • Unsuccessful project money and unsuccessful projects count include failed, canceled, and suspended projects. 
    • Success rate is calculated by the total number of successful projects divided by the total number of closed (i.e. not-live) projects, which includes canceled, suspended, etc.
    • We'd like not to distinguish between failed/canceled/suspended. We would just call them Unsuccessful.

Audience, Participation & Demographics

TRG (List of Audience Members)
Detroit Audience List - data set not yet collected; this data set might not be acquired in time for Detroit launch
Link to file - Bay Area Audience List
  • This data set is a list of audience members for a subset of non-profit organizations in the instance region.
  • A similar data set is already integrated into BACAM (audience_members and audience_organizations tables), though there are differences between the old source and the new source.
  • Important: this data source does *not* provide EINs for the organizations that each audience member is attached to; therefore, we will have to create an import script that attempts a name lookup and, if none is found, the organizations will need to be manually matched with organizations in our database. (Name lookup to match an import source w/ an entity in the database will be required in other places in the application as well, so it is probably worth building.)
  • If the Detroit audience list is collected, the fields may be slightly different from the Bay Area list.
  • Each HHID belongs to a single audience member. The front end should be able to display trends in audience crossover between non-profit organizations. (If audience member 11111 patronized organizations X & Y and audience member 22222 patronized organizations X, Y and Z, then organizations X & Y share an audience member in common.)

NEA (Federal arts grants)
Link to CSV - Grants from NEA to Detroit non-profit organizations
Grants from NEA to Bay Area non-profit organizations - waiting for cleaning
  • These grants are already imported for Detroit instance.
  • All grants from this source should have the grantmaker_id set as the NEA (National Endowment for the Arts).
  • Question: how are current grants matched w/ organizations in front end code by Stamen? (There is no EIN for grantees in this data set.)
  • We don't have EINs for the organizations in the NEA and NASAA spreadsheets. Somehow, we need to figure out if we already have these organizations imported from Guidestar, and if we do, connect their grant information from NEA and NASAA with their organization information from Guidestar. I suggest a name lookup. Here's one idea for how this could be done:
  1. For each grant in NEA/NASAA, look at the grantee name (NEA="Grantee" and NASAAA="ApplName").
  2. Try to match the name to an organization already in our database. If there's a match, set "grantee_id" to the correct grantee.
  3. For organizations that don't match, try to find a name match via the Guidestar API.
  4. If no match can be found, we'll need to figure out a way for an intern to match the grantee organizations w/ an organization EIN. Maybe a spreadsheet export of all grants that have not been matched to organizations that the intern can fill in?
